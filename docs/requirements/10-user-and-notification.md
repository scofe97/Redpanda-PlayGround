# 사용자 및 알림

## TPS 원본 — 마인드맵에서 무엇을 하는가

TPS에서 사용자 관리와 알림은 다음을 포함한다:

- **사용자 기본**: 계정 관리, 프로필
- **빌드배포 권한**: 누가 어떤 파이프라인을 실행/조회할 수 있는지
- **알림**: 파이프라인 완료/실패 시 알림 발송
- **결재**: 워크플로우의 승인 절차 (PoC 범위 외)
- **툴체인 관리**: 트렁본 수정 버전 (참고용)

---

## PoC 목표 — Playground에서 무엇을 검증하는가

1. **기본 사용자 모델**: 파이프라인/티켓의 생성자/실행자를 추적하는 최소한의 사용자 모델을 구현한다
2. **권한 기본 구조**: 역할 기반 접근 제어(RBAC)의 기본 뼈대를 만든다. 모든 기능을 세밀하게 제어하지는 않지만, 관리자/운영자/조회자 수준의 역할 분리를 검증한다
3. **파이프라인 이벤트 알림**: 파이프라인 완료/실패 시 알림을 발송하는 구조를 구현한다. 기존 SSE는 브라우저 접속 중인 사용자에게만 전달되므로, 오프라인 알림(이메일/웹훅) 경로를 추가한다

결재는 PoC 범위 외이며, 02-workflow의 결재 스텁(REQ-02-006)에서 최소한만 다룬다.

## 요구사항

### REQ-10-001: 사용자 기본 모델 [P1]

파이프라인/티켓의 소유자와 실행자를 추적하는 최소 사용자 모델을 구현한다.

- 사용자 ID, 이름, 이메일
- 역할: ADMIN / OPERATOR / VIEWER
- 사용자 CRUD (PoC: 초기 데이터 Flyway 마이그레이션으로 삽입)
- 티켓 생성 시 `createdBy`, 파이프라인 실행 시 `executedBy` 기록
- 인증은 PoC에서 단순화: 요청 헤더에 사용자 ID 전달 (Spring Security 미사용)

### REQ-10-002: 역할 기반 접근 제어 [P2]

역할에 따라 API 접근을 제한하는 기본 구조를 만든다.

| 역할 | 파이프라인 | 티켓 | 미들웨어/환경 | 사용자 |
|------|----------|------|-------------|--------|
| ADMIN | 전체 | 전체 | 생성/수정/삭제 | 관리 |
| OPERATOR | 생성/실행 | 생성/실행 | 조회 | - |
| VIEWER | 조회 | 조회 | 조회 | - |

- 인터셉터 또는 AOP 기반 권한 체크
- 권한 부족 시 403 Forbidden
- PoC에서는 어노테이션 기반 선언적 체크 (예: `@RequireRole(OPERATOR)`)

### REQ-10-003: 파이프라인 이벤트 알림 [P1]

파이프라인/워크플로우 이벤트 발생 시 관련 사용자에게 알림을 발송한다.

- 알림 이벤트: 파이프라인 완료/실패, 워크플로우 결재 대기, 결재 완료
- 알림 채널:
  - SSE (기존 — 브라우저 접속 중)
  - 웹훅 (HTTP POST — Slack/Teams 등 외부 서비스 연동)
  - 이메일 (P2 — SMTP 연동)
- 사용자별 알림 설정: 어떤 이벤트를 어떤 채널로 받을지
- 기존 audit.events 토픽을 알림 소스로 활용

### REQ-10-004: 알림 어댑터 [P1]

알림 채널별 어댑터를 정의한다.

```
NotificationAdapter
├── send(userId, event, message) → void
```

- SseNotificationAdapter: 기존 SseEmitterRegistry 활용
- WebhookNotificationAdapter: 설정된 URL로 HTTP POST
- EmailNotificationAdapter: SMTP 연동 (P2)
- 사용자별 알림 설정에 따라 어댑터 선택

## 설계 고민 포인트

### 인증 시스템의 범위

PoC에서 Spring Security를 도입하면 복잡도가 크게 올라간다. JWT 토큰 발행/검증, 로그인 API, 세션 관리 등이 필요해지는데, 이는 PoC의 핵심 검증 대상(이벤트 기반 파이프라인)과 관련이 없다. 대안으로 요청 헤더에 `X-User-Id`를 넣는 가벼운 방식을 사용하고, 인터셉터에서 이를 읽어 SecurityContext에 넣는 것으로 충분하다. 프로덕션 전환 시 헤더 인증을 JWT로 교체하면 된다.

### 알림과 기존 SSE의 관계

현재 SSE는 파이프라인 실행 상태를 실시간으로 브라우저에 푸시하는 용도다. 이것과 "알림"은 다른 개념이다. SSE는 "진행 중인 작업의 상태 스트림"이고, 알림은 "완료/실패 같은 이벤트의 통지"다. 두 가지를 같은 인프라(SSE)로 전달할 수도 있지만, 브라우저를 닫은 사용자에게는 SSE가 전달되지 않으므로 별도 채널(웹훅/이메일)이 필요하다. 알림 발행은 기존 audit.events 토픽의 컨슈머로 구현하면 새로운 토픽을 만들 필요가 없다.

### 결재와 알림의 관계

결재는 PoC 범위 외이지만, 워크플로우의 결재 스텁(REQ-02-006)에서 "결재 대기" 상태가 발생하면 알림이 필요하다. 이 경우 워크플로우 엔진이 "결재 대기" 이벤트를 발행하고, 알림 컨슈머가 이를 수신하여 해당 사용자에게 통지하는 구조다. 결재 시스템이 없더라도 알림 인프라는 독립적으로 동작할 수 있다.
