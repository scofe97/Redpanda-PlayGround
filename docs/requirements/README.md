# 요구사항 문서

> **권위 있는 소스가 `specs/domains/`로 이동했습니다.** 이 디렉토리의 파일은 원본 기록으로 유지되지만, 최신 스펙(수용 기준, 상태 갱신, Phase 3 신규 REQ 포함)은 `specs/domains/` 를 참조하세요.

TPS CI/CD 플랫폼의 마인드맵을 참고하여, Redpanda Playground PoC에서 검증할 요구사항을 도메인별로 정리한 문서 모음이다. TPS를 그대로 복제하는 것이 아니라 핵심 개념을 추출하고 내 방식대로 재해석하여 구현하는 것이 목적이다.

---

## 3개 독립 개념의 관계

```
[티켓]                  [워크플로우]                    [파이프라인]
 업무코드(팀 코드) ──→ 컴포넌트 조합 ──→ 빌드/배포 → 파이프라인 참조
 파이프라인 그룹         (빌드/배포/결재)    결재 → 워크플로우 레벨
```

- **티켓**: 배포 요청 단위. 업무코드를 부여하면 워크플로우가 결정된다
- **워크플로우**: 업무코드별 프로세스 템플릿. 빌드/배포/결재 컴포넌트 조합을 정의한다
- **파이프라인**: 독립적인 Job 실행 체인. 워크플로우가 참조할 수도 있고, 단독 실행도 가능하다

> 현재 PoC 상태: 티켓 → 바로 파이프라인 (업무코드/워크플로우 없음, 파이프라인이 티켓에 종속)

## 멀티모듈 구조

독립 배포 단위는 **2개**다. 나머지 도메인은 이 2개 모듈 안의 패키지로 시작하고, 규모가 커지면 분리한다. 요구사항 문서 10개는 도메인 기능 정의이지 모듈 구조가 아니다.

| 모듈 | 포함 도메인 | 비고 |
|------|-----------|------|
| **pipeline** | 파이프라인(03), 빌드 Job(04), 배포 Job(06), 미들웨어(08), 배포 환경(09), 결과물(07) | 실행 엔진 + Job + 인프라 |
| **ticket** | 티켓(01), 워크플로우(02), 사용자/알림(10) | 업무 요청 + 프로세스 |
| **common** | 공통 인프라 (Outbox, 멱등성, DTO) | 기존 유지 |

## 유스케이스

### UC-1: 파이프라인 단독 실행 (티켓 없이)

```
관리자: 지원 도구 등록(Jenkins/Harbor/GitLab) + 배포 환경 등록(K8S 클러스터)
      → 미들웨어 프리셋 생성 ("preset-team-a" = Jenkins-A + ArgoCD + Harbor)
사용자: 파이프라인 생성(자유형) → Job 1:빌드(Git+Script) + Job 2:배포(K8S+매니페스트)
      → 프리셋 "preset-team-a" 참조
사용자: 파이프라인 실행(파라미터: IMAGE_TAG=v1.2.3)
시스템: Job 1 → CI 어댑터(Jenkins) 위임 → Redpanda 경유 → 완료 → Harbor push
시스템: Job 2 → CD 어댑터(ArgoCD) 또는 K8S API로 매니페스트 적용
사용자: SSE 실시간 확인 → 완료
```

### UC-2: 티켓 → 워크플로우 → 파이프라인

```
관리자: 업무코드 "TEAM-A" 정의 → 워크플로우(빌드+배포+결재) 생성
      → 워크플로우가 파이프라인 "team-a-deploy" 참조
      → 업무코드 "TEAM-A"의 기본 프리셋 = "preset-team-a"
사용자: 티켓 생성(업무코드: TEAM-A, 소스: GIT) → 워크플로우 자동 결정
시스템: 워크플로우 Step 1 → 파이프라인 실행 (프리셋 주입, UC-1과 동일 흐름)
시스템: 워크플로우 Step 2 → 결재 대기 (PoC: 자동 승인 또는 스킵)
사용자: 결재 승인 → 티켓 상태: DEPLOYED
```

### UC-3: 빌드 반입 + VM 배포

```
사용자: 파이프라인 생성 → Job 1:빌드(반입, harbor.io/my-app:v1.0) + Job 2:배포(VM, prod-server-01)
시스템: Job 1 → Harbor 이미지 존재 확인만 (빌드 스킵)
시스템: Job 2 → 배포 환경에서 VM SSH 정보 조회 → 원격 스크립트 실행
```

### UC-4: 아티팩트 CRUD

```
사용자: Harbor 이미지 조회 / Nexus JAR 검색 / Minio 파일 목록
사용자: Nexus에 JAR 직접 업로드 / Harbor에 이미지 push (빌드 결과 자동 연계)
사용자: 태그 변경, 메타데이터 수정
```

### UC-5: 미들웨어 교체 (Jenkins → GoCD)

```
관리자: GoCD 등록(역할: CI/CD Tool)
관리자: 프리셋 "preset-team-a" 수정 → CI: Jenkins-A → GoCD
시스템: 해당 프리셋을 참조하는 모든 파이프라인이 자동으로 GoCD 사용
       → 파이프라인 개별 수정 불필요 (프리셋 참조이므로)
```

## 설계 원칙

1. **파이프라인 독립성**: 파이프라인은 티켓/워크플로우 없이 단독 존재/실행이 가능하다
2. **미들웨어 프리셋**: 역할별 도구 조합을 프리셋으로 묶고, 파이프라인이 프리셋을 참조한다
3. **CI/CD 추상화**: Jenkins에 한정하지 않고 CiAdapter/CdAdapter 인터페이스로 대체 가능하게 한다
4. **결재는 워크플로우 레벨**: 파이프라인은 실행 체인만 관리하고, 결재는 워크플로우에서 처리한다
5. **점진적 모듈 분리**: pipeline/ticket 2개 모듈로 시작하고, 도메인은 패키지로 구분한다
