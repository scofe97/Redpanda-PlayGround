# 티켓 관리

## TPS 원본 — 마인드맵에서 무엇을 하는가

TPS에서 티켓은 배포 요청의 시작점이다. 각 티켓에는 업무코드(팀 단위 코드명)가 부여되고, 업무코드가 워크플로우를 결정한다. 워크플로우는 빌드/배포/결재 등의 컴포넌트 조합으로 구성되며, 이 중 빌드/배포 컴포넌트가 파이프라인을 참조하여 실행된다. 티켓은 파이프라인 그룹에 속하지만, 파이프라인 자체는 티켓에 종속되지 않는다.

핵심 관계:
- 업무코드 → 워크플로우 자동 결정
- 워크플로우 → 빌드/배포 컴포넌트 → 파이프라인 참조
- 결재 → 워크플로우 레벨에서 처리

---

## PoC 목표 — Playground에서 무엇을 검증하는가

1. **업무코드 개념 도입**: 티켓에 업무코드를 부여하고, 업무코드로 워크플로우를 결정하는 흐름을 검증한다
2. **파이프라인 독립성 확보**: 현재 파이프라인이 티켓에 종속되어 있는 구조를 분리한다. 티켓 없이도 파이프라인이 존재하고 실행될 수 있어야 한다
3. **티켓 → 워크플로우 → 파이프라인 3단 연계**: 티켓 생성 시 업무코드 기반으로 워크플로우가 결정되고, 워크플로우가 파이프라인을 참조하여 실행하는 전체 흐름을 구현한다

## 현재 구현 상태

| 항목 | 상태 | 비고 |
|------|------|------|
| Ticket CRUD | 구현됨 | 생성/조회/수정/삭제 |
| TicketSource | 구현됨 | GIT/NEXUS/HARBOR 소스 유형 |

| 업무코드 | 미구현 | 티켓에 업무코드 필드 없음 |
| 워크플로우 연계 | 미구현 | 티켓 → 바로 파이프라인 |
| 파이프라인 독립성 | 미구현 | 파이프라인이 티켓에 종속 |

## 요구사항

### REQ-01-001: 업무코드 필드 추가 [P0]

티켓 엔티티에 업무코드(businessCode) 필드를 추가한다. 업무코드는 팀 단위 코드명으로, 티켓 생성 시 필수 입력이다. 업무코드를 기반으로 해당 팀이 따를 워크플로우가 결정된다.

- 업무코드는 문자열 (예: "TEAM-A", "PLATFORM", "INFRA")
- 업무코드 목록은 별도 테이블로 관리 (BusinessCode 엔티티)
- 티켓 생성 API에서 유효한 업무코드인지 검증

### REQ-01-002: 워크플로우 참조 [P0]

티켓이 업무코드를 통해 워크플로우를 참조하는 구조를 만든다. 티켓 생성 시 업무코드 → 워크플로우 매핑을 조회하여 어떤 프로세스를 따를지 자동 결정한다.

- 하나의 업무코드는 하나의 워크플로우에 매핑
- 워크플로우 매핑이 없는 업무코드는 "기본 워크플로우"(빌드+배포만) 적용
- 티켓 조회 시 연결된 워크플로우 정보 포함

### REQ-01-003: 파이프라인 독립성 확보 [P0]

기존의 "티켓 생성 → 파이프라인 자동 생성" 종속 관계를 끊는다. 파이프라인은 독립적으로 존재하며, 워크플로우를 통해 간접 참조된다.

- 티켓 생성이 파이프라인 생성을 트리거하지 않음
- 워크플로우의 빌드/배포 컴포넌트가 기존 파이프라인을 참조
- 파이프라인은 티켓 없이 단독 실행 가능 (UC-1 참조)

### REQ-01-004: 업무코드 CRUD [P1]

업무코드를 관리하는 API를 제공한다.

- 업무코드 생성/조회/수정/삭제
- 업무코드별 기본 워크플로우 매핑 설정
- 업무코드별 기본 미들웨어 프리셋 매핑 설정 (편의 기능)

### REQ-01-005: 티켓 상태 확장 [P1]

워크플로우 도입에 따라 티켓 상태를 확장한다.

- 기존: DRAFT → READY → DEPLOYING → DEPLOYED/FAILED
- 확장: DRAFT → READY → **WORKFLOW_IN_PROGRESS** → DEPLOYING → **APPROVAL_PENDING** → DEPLOYED/FAILED/REJECTED
- WORKFLOW_IN_PROGRESS: 워크플로우 실행 중 (빌드/배포 진행)
- APPROVAL_PENDING: 결재 대기 (워크플로우에 결재 컴포넌트가 있을 때)

### REQ-01-006: 티켓-파이프라인 실행 이력 [P2]

티켓이 트리거한 파이프라인 실행 이력을 추적한다.

- 하나의 티켓이 여러 번 파이프라인을 실행할 수 있음 (재시도)
- 각 실행의 결과(성공/실패), 시간, 파이프라인 ID를 기록
- 이력 조회 API

## 설계 고민 포인트

### 업무코드와 워크플로우의 1:1 vs 1:N

TPS에서는 업무코드 하나에 워크플로우 하나가 매핑된다. PoC에서도 1:1로 시작하되, 추후 "같은 팀이지만 상황에 따라 다른 프로세스"가 필요하면 1:N으로 확장할 수 있는 여지를 남겨둔다. 초기에는 1:1이 단순하고 충분하다.

### 파이프라인 독립성을 확보하면서 티켓과의 연결 추적

파이프라인을 독립시키면 "이 파이프라인 실행이 어떤 티켓에서 트리거됐는지" 추적이 필요하다. 파이프라인 실행 시 `triggerSource` (TICKET/MANUAL/WORKFLOW) + `triggerId`를 기록하는 방식이 자연스럽다. 파이프라인 엔티티 자체에는 티켓 참조가 없고, 실행(Execution) 레벨에서만 추적한다.

### 업무코드 없이 티켓을 생성할 수 있는가

PoC 초기 단계에서는 업무코드 없이도 티켓을 생성할 수 있어야 한다. 기존 기능이 깨지지 않도록 업무코드는 optional로 시작하고, 워크플로우 연계가 안정화되면 필수로 전환하는 마이그레이션 전략이 필요하다.
