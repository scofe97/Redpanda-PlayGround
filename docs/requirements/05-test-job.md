# 테스트 Job

## TPS 원본 — 마인드맵에서 무엇을 하는가

TPS에서 자동화 테스트 Job은 빌드 Job의 Git 정보와 Base Image를 공유하면서, JUnit 테스트 실행 + SonarQube 정적 분석을 수행한다. 테스트 결과는 파이프라인 실행 이력에 첨부되고, 품질 기준 미달 시 파이프라인을 실패 처리한다.

---

## PoC 결정 — 생략

테스트 Job은 PoC 범위에서 **생략**한다. 이유는 다음과 같다:

1. PoC의 핵심 검증 대상은 "이벤트 기반 파이프라인 오케스트레이션"이지 "CI 품질 게이트"가 아니다
2. JUnit/SonarQube 연동은 빌드 Job의 CI 어댑터 확장으로 자연스럽게 추가할 수 있다
3. 테스트 Job의 구조는 빌드 Job과 거의 동일하다 (CI 어댑터에 다른 스크립트를 전달할 뿐)

---

## TPS 참고 기록

추후 구현할 경우를 위해 TPS의 테스트 Job 구조를 기록해둔다.

### 테스트 유형

| 유형 | 도구 | 설명 |
|------|------|------|
| 단위 테스트 | JUnit | 빌드 스크립트 내에서 실행 |
| 정적 분석 | SonarQube | 코드 품질/보안 취약점 스캔 |
| 통합 테스트 | (프로젝트별) | 외부 의존성 포함 테스트 |

### 빌드 Job과의 공유 항목

- Git URL, 브랜치/태그 (동일 소스)
- Base Image (동일 빌드 환경)
- credentialId (동일 인증 정보)

### 품질 게이트

- SonarQube 품질 기준 통과 여부로 파이프라인 진행/중단
- 테스트 커버리지 기준 설정 (예: 80% 이상)
- 테스트 실패 시 파이프라인 FAILED 처리

---

## 구현 시 고려사항

테스트 Job을 추가할 때는 Job 타입에 `TEST`를 추가하고, 빌드 Job과 유사한 config 스키마를 사용하되 output 대신 testReport를 반환하는 구조가 될 것이다. CI 어댑터가 이미 추상화되어 있다면, 테스트 실행도 동일한 어댑터를 통해 CI 미들웨어에 위임할 수 있다.
