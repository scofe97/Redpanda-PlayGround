# 워크플로우 관리

## TPS 원본 — 마인드맵에서 무엇을 하는가

TPS에서 워크플로우는 업무코드별 프로세스 템플릿이다. 워크플로우는 컴포넌트들의 조합으로 구성되며, 각 컴포넌트는 빌드, 배포, 빌드/배포, 자동화테스트, 결재 중 하나다. 빌드/배포 컴포넌트는 파이프라인을 참조하여 실행하고, 결재 컴포넌트는 워크플로우 레벨에서 승인 절차를 관리한다. 파이프라인이 "어떻게 실행할지"를 정의한다면, 워크플로우는 "무엇을 어떤 순서로 진행할지"를 정의한다.

---

## PoC 목표 — Playground에서 무엇을 검증하는가

1. **컴포넌트 조합 패턴**: 빌드/배포/결재를 컴포넌트 단위로 조합하는 구조를 검증한다. 워크플로우가 파이프라인과 독립적으로 프로세스 흐름을 정의하는 개념이 핵심이다
2. **워크플로우-파이프라인 연계**: 워크플로우의 빌드/배포 컴포넌트가 기존 파이프라인을 참조하여 실행하는 구조를 구현한다
3. **결재 흐름 스텁**: 실제 결재 시스템을 구현하지는 않지만, 워크플로우에서 "결재 대기 → 승인/거절" 상태 전이가 가능한 스텁을 만든다

---

## 현재 구현 상태

워크플로우는 현재 구현되지 않았다. 티켓이 파이프라인을 직접 트리거하는 구조이며, 중간에 컴포넌트 조합이나 결재 개념이 없다.

## 요구사항

### REQ-02-001: 워크플로우 엔티티 정의 [P0]

워크플로우를 정의하는 엔티티를 설계한다. 워크플로우는 이름, 설명, 컴포넌트 목록으로 구성된다.

- 워크플로우 ID, 이름, 설명
- 컴포넌트 목록 (순서가 있는 리스트)
- 활성 상태 (ACTIVE/INACTIVE)
- 생성/수정 시간

### REQ-02-002: 컴포넌트 타입 정의 [P0]

워크플로우를 구성하는 컴포넌트 타입을 정의한다. 각 컴포넌트 타입별로 참조해야 할 대상이 다르다.

| 컴포넌트 타입 | 참조 대상 | 설명 |
|-------------|----------|------|
| BUILD | 파이프라인 ID | 빌드 파이프라인 실행 |
| DEPLOY | 파이프라인 ID | 배포 파이프라인 실행 |
| BUILD_AND_DEPLOY | 파이프라인 ID | 빌드+배포 통합 파이프라인 |
| APPROVAL | 승인자 정보 | 결재 대기 (PoC: 자동 승인 스텁) |

- 하나의 워크플로우에 같은 타입의 컴포넌트가 여러 개일 수 있음 (예: 빌드 → 결재 → 배포)
- 각 컴포넌트는 순서(order) 필드를 가짐

### REQ-02-003: 워크플로우 CRUD [P0]

워크플로우를 관리하는 API를 제공한다.

- 워크플로우 생성: 이름 + 컴포넌트 리스트
- 워크플로우 조회: 상세(컴포넌트 포함) + 목록
- 워크플로우 수정: 컴포넌트 추가/제거/순서 변경
- 워크플로우 삭제: 참조 중인 업무코드가 없을 때만

### REQ-02-004: 업무코드-워크플로우 매핑 [P0]

업무코드와 워크플로우의 매핑을 관리한다. 티켓 생성 시 업무코드를 입력하면, 매핑된 워크플로우가 자동으로 결정된다.

- 업무코드 1개 → 워크플로우 1개 (1:1 매핑)
- 매핑 생성/조회/수정/삭제
- 매핑이 없는 업무코드는 기본 워크플로우 적용

### REQ-02-005: 워크플로우 실행 엔진 [P1]

워크플로우를 실행하는 엔진을 구현한다. 워크플로우 실행은 컴포넌트를 순서대로 처리하는 것이다.

- 빌드/배포 컴포넌트: 참조된 파이프라인 실행을 트리거하고 완료를 대기
- 결재 컴포넌트: 결재 대기 상태로 전환, 승인/거절 이벤트 수신
- 컴포넌트 간 상태 전이: PENDING → IN_PROGRESS → COMPLETED/FAILED
- 실패 시 워크플로우 전체 상태를 FAILED로 전환

### REQ-02-006: 결재 스텁 [P2]

실제 결재 시스템을 구현하지 않되, 워크플로우에서 결재 컴포넌트가 작동하는 구조를 스텁으로 만든다.

- 결재 요청 생성 (워크플로우 실행 중 결재 컴포넌트 도달 시)
- 결재 승인/거절 API (수동 호출)
- PoC 옵션: 자동 승인 모드 (설정으로 on/off)
- 결재 결과에 따라 워크플로우 다음 단계 진행 또는 중단

### REQ-02-007: 워크플로우 실행 이력 [P2]

워크플로우 실행 내역을 추적한다.

- 워크플로우 실행 ID, 티켓 ID, 워크플로우 ID
- 각 컴포넌트 실행 상태 + 시작/종료 시간
- 파이프라인 참조 컴포넌트: 파이프라인 실행 ID 연결

## 설계 고민 포인트

### 워크플로우 실행과 파이프라인 실행의 관계

워크플로우의 빌드/배포 컴포넌트가 파이프라인을 "참조"한다고 했을 때, 실행 시점에 파이프라인의 어떤 것을 호출하는가가 핵심이다. 워크플로우 엔진이 파이프라인 실행 API를 호출하고, 파이프라인 완료 이벤트를 수신하여 다음 컴포넌트로 진행하는 이벤트 기반 연계가 자연스럽다. 동기 호출은 워크플로우 엔진을 블로킹하므로 피한다.

### 결재와 파이프라인의 레벨 분리

결재를 파이프라인 레벨이 아닌 워크플로우 레벨에서 처리하는 이유는 관심사 분리 때문이다. 파이프라인은 "Job을 순서대로 실행하는 기계"이고, 결재는 "비즈니스 프로세스의 승인 절차"다. 파이프라인에 결재를 넣으면 파이프라인의 독립성이 깨진다. 예를 들어 같은 파이프라인을 테스트 환경에서는 결재 없이, 프로덕션에서는 결재 후 실행하고 싶을 때, 결재 유무가 워크플로우 레벨에 있어야 유연하다.

### 워크플로우가 없는 단순 실행

모든 파이프라인 실행이 워크플로우를 거쳐야 하는 것은 아니다. UC-1처럼 파이프라인을 직접 실행할 수 있어야 한다. 워크플로우는 "티켓 기반 업무 프로세스"가 필요한 경우에만 개입한다. 파이프라인 API는 워크플로우를 모르고, 워크플로우 엔진이 파이프라인 API를 호출하는 단방향 의존이 올바르다.
