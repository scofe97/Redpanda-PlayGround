# 배포 환경

## TPS 원본 — 마인드맵에서 무엇을 하는가

TPS에서 환경 설정은 자원관리와 버전관리로 나뉜다:

- **자원관리**: 서버관리(VM 호스트 정보), 클러스터관리(K8S 클러스터 정보)
- **버전관리**: 저장소관리(Git 리포지토리 정보)

배포 Job(06)이 "어디에 배포할지"를 결정할 때, 이 도메인에서 대상 환경 정보를 참조한다. 환경은 미들웨어(도구)와 별도로 관리되는 "배포 목적지" 도메인이다.

---

## PoC 목표 — Playground에서 무엇을 검증하는가

1. **서버/클러스터 관리**: VM 서버와 K8S 클러스터 정보를 등록하고 배포 Job에서 참조하는 구조를 구현한다
2. **환경과 미들웨어의 관심사 분리**: 미들웨어(08)는 "어떤 도구로"를 관리하고, 배포 환경(09)은 "어디에"를 관리한다. 두 도메인이 독립적으로 동작하되, 배포 시 함께 사용되는 관계를 검증한다

---

## 현재 구현 상태

배포 환경 도메인은 현재 구현되지 않았다. 배포 Job이 환경 정보를 참조하지 않고, Jenkins Job에 하드코딩된 설정을 사용한다.

---

## 요구사항

### REQ-09-001: VM 서버 엔티티 [P0]

VM 서버 정보를 관리하는 엔티티를 정의한다.

- 서버 ID, 이름, 설명
- 호스트(IP 또는 도메인), SSH 포트
- 인증 방식: SSH 키 / 패스워드
- 인증 정보 참조 (credentialId — 평문 저장 금지)
- 배포 경로 (기본 디렉토리)
- 상태: ACTIVE / INACTIVE
- 태그/라벨 (환경별: dev/staging/prod)

### REQ-09-002: K8S 클러스터 엔티티 [P0]

K8S 클러스터 정보를 관리하는 엔티티를 정의한다.

- 클러스터 ID, 이름, 설명
- API Server URL
- 인증 방식: kubeconfig / ServiceAccount Token
- 인증 정보 참조 (credentialId)
- 기본 네임스페이스
- 상태: ACTIVE / INACTIVE
- 태그/라벨 (환경별: dev/staging/prod)

### REQ-09-003: 환경 CRUD [P0]

VM 서버와 K8S 클러스터를 통합 관리하는 API를 제공한다.

- VM 서버 CRUD: 등록/조회/수정/삭제
- K8S 클러스터 CRUD: 등록/조회/수정/삭제
- 통합 목록 조회: 환경 유형(VM/K8S) 필터, 태그 필터
- 환경 연결 테스트: SSH 접속 테스트 / K8S API 서버 접근 테스트

### REQ-09-004: 배포 Job 연계 [P1]

배포 Job(06)이 환경 엔티티를 참조하여 배포 대상을 결정하는 구조를 구현한다.

- 배포 Job config에 `environmentId` 필드
- 파이프라인 실행 시 environmentId → 환경 엔티티 조회 → 접속 정보 해석
- VM 배포: 서버 SSH 정보 사용
- K8S 배포: 클러스터 API 서버 + 인증 정보 사용

### REQ-09-005: 인증 정보 관리 [P2]

환경 접속에 필요한 인증 정보를 안전하게 관리한다.

- Credential 엔티티: ID, 이름, 타입(SSH_KEY/PASSWORD/TOKEN/KUBECONFIG), 값(암호화)
- 환경 엔티티에서 credentialId로 참조
- 값은 AES 등으로 암호화 저장 (PoC: 간단한 암호화 또는 환경변수 참조)
- 조회 API에서 값을 마스킹하여 반환

## 설계 고민 포인트

### VM 서버와 K8S 클러스터를 하나의 엔티티로 통합할 것인가

두 가지 접근이 가능하다:

1. **통합 엔티티**: Environment(type: VM/K8S) + config JSON으로 유형별 정보 구분. 단순하지만 config 스키마가 유형마다 다르다
2. **분리 엔티티**: VmServer, K8sCluster 별도 테이블. 타입 안전하지만 API가 분리된다

PoC에서는 통합 엔티티로 시작한다. `environmentType: VM | K8S`와 config JSON으로 구분하되, config의 필수 필드를 유형별로 검증한다. 규모가 커지면 분리 엔티티로 전환할 수 있다.

### 인증 정보의 보안 수준

PoC이므로 프로덕션 수준의 Vault 연동까지는 불필요하다. 하지만 평문 저장은 피해야 한다. DB에 AES-256으로 암호화 저장하고, 키는 환경변수에서 로드하는 수준이 적절하다. Credential 엔티티를 분리해두면, 추후 HashiCorp Vault나 AWS Secrets Manager로 교체하기 쉽다.

### 환경과 프리셋의 관계

미들웨어 프리셋은 "도구 조합"이고, 환경은 "배포 목적지"다. 프리셋에 ArgoCD가 있고, 환경에 K8S 클러스터가 있으면, 배포 Job이 "프리셋에서 ArgoCD 정보 → 환경에서 K8S 클러스터 정보"를 각각 조회한다. 두 도메인은 직접 참조하지 않고, 배포 Job 실행 시점에 합류한다.
