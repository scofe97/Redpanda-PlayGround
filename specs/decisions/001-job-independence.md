# ADR-001: Job을 파이프라인에서 독립 엔티티로 분리

## 상태

Accepted

## 맥락

초기 설계에서 Job은 파이프라인에 종속된 개념이었다. 파이프라인이 생성될 때 고정된 3단계(소스 체크아웃 → 빌드 → 배포)가 함께 생성되었고, Job은 파이프라인 외부에서 단독으로 존재할 수 없었다.

이 구조에서는 동일한 빌드 로직을 여러 파이프라인에서 재사용할 방법이 없었다. "서비스 A 빌드"와 "서비스 B 빌드"가 거의 같은 설정이더라도 각 파이프라인마다 별도 Job 레코드를 만들어야 했다. 빌드 스크립트 변경이 생기면 모든 파이프라인의 Job을 일일이 수정해야 하는 유지보수 문제가 명확했다.

또한 고정 3단계 구조로는 "빌드만 하는 파이프라인"이나 "검증 → 배포"처럼 단계를 자유롭게 조합하는 요구를 표현할 수 없었다.

## 결정

`pipeline_job`을 파이프라인과 독립된 별도 테이블로 분리한다. 파이프라인과 Job은 `pipeline_job_mapping` 테이블을 통해 다대다 관계로 연결된다. 하나의 Job은 여러 파이프라인에서 참조할 수 있고, 파이프라인은 임의 개수의 Job을 임의 순서로 조합할 수 있다.

테이블 구조는 다음과 같다.

```
pipeline_job          pipeline_job_mapping       pipeline
─────────────         ────────────────────       ────────
id                    pipelineId (FK)            id
name                  jobId (FK)                 name
jobType               executionOrder             ...
config (JSON)         ...
...
```

## 근거

Job을 독립 엔티티로 분리하면 세 가지 이점을 얻는다.

첫째, 재사용성이다. "Maven 빌드 Job"을 한 번 정의하면 여러 파이프라인에서 참조할 수 있다. 빌드 파라미터 변경 시 Job 레코드 하나만 수정하면 해당 Job을 사용하는 모든 파이프라인에 반영된다.

둘째, 조합 유연성이다. 파이프라인은 필요한 Job만 골라서 원하는 순서로 조합한다. "빌드만", "배포만", "빌드 → 검증 → 배포" 같은 다양한 조합을 하나의 구조로 표현할 수 있다.

셋째, 독립 테스트 가능성이다. Job 단위로 실행 결과를 추적하고 테스트할 수 있어서 어떤 Job이 실패했는지 명확히 식별된다.

## 대안

**파이프라인 내 인라인 Job 정의**: 파이프라인 엔티티의 JSON 컬럼에 Job 설정을 포함하는 방식이다. 단순하지만 Job 재사용이 불가능하고, 같은 설정을 여러 파이프라인에 복사해야 한다. 변경 시 모든 파이프라인을 찾아서 수정해야 하므로 유지보수 비용이 높다.

**고정 스텝 파이프라인**: 현재 구현 방식으로, 항상 3단계가 고정된다. TPS의 실제 요구사항인 "빌드만", "배포만" 파이프라인을 표현할 수 없어서 폐기했다.

## 영향

- DB 스키마에 `pipeline_job`과 `pipeline_job_mapping` 테이블이 추가된다
- 파이프라인 생성 API가 Job ID 목록을 받는 방식으로 변경된다
- 기존 하드코딩된 3단계 파이프라인 생성 로직을 제거해야 한다
- Job 단위 실행 추적이 가능해져 DAG 엔진(ADR-002)의 기반이 된다
