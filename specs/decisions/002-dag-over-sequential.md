# ADR-002: 순차 실행 대신 DAG 병렬 실행 선택

## 상태

Accepted

## 맥락

ADR-001에서 Job을 독립 엔티티로 분리한 뒤, "파이프라인 내 Job들을 어떤 순서로 실행할지"를 결정해야 했다.

가장 단순한 방법은 `executionOrder` 숫자로 순서를 고정하는 것이다. order 1 완료 → order 2 시작 → order 3 시작 식의 완전 순차 실행이다.

그런데 TPS의 실제 배포 패턴을 분석하면, 독립적으로 병렬 실행 가능한 작업이 많다. 예를 들어 서비스 A 빌드와 서비스 B 빌드는 서로 의존하지 않으므로 동시에 실행할 수 있고, 두 빌드가 모두 완료된 후에야 통합 배포를 시작해야 한다. 이러한 관계를 순차 실행만으로는 표현할 수 없다. 순차 실행 강제 시 병렬 가능한 작업도 직렬화되어 전체 파이프라인 시간이 불필요하게 길어진다.

또한 "A 완료 후 B와 C 동시 시작, B와 C 모두 완료 후 D 시작"과 같은 팬아웃/팬인 패턴도 실무에서 자주 등장한다.

## 결정

`pipeline_job_dependency` 테이블을 도입하여 Job 간 선행 관계를 명시적으로 정의한다. 실행 엔진은 Kahn's Algorithm(위상 정렬)으로 실행 순서를 계산하고, 의존 Job이 모두 완료된 Job들을 병렬로 실행한다.

```
pipeline_job_dependency
───────────────────────
pipelineId
jobId          (이 Job이 실행되려면)
dependsOnJobId (이 Job이 먼저 완료되어야 한다)
```

의존 관계가 없는 Job들은 파이프라인 시작 시 동시에 실행된다. 순환 의존이 있으면 파이프라인 저장 시 유효성 검사에서 거부한다.

## 근거

DAG 구조를 선택한 이유는 세 가지다.

첫째, TPS 실제 사용 패턴에 부합한다. 멀티 서비스 배포에서 독립 빌드를 병렬화하면 전체 배포 시간을 단축할 수 있다.

둘째, 표현력이 충분하다. 순차 실행은 DAG의 특수 케이스(모든 Job이 단일 체인)이므로, DAG 구조를 도입해도 기존 순차 실행 파이프라인을 그대로 표현할 수 있다. 기능이 추가되는 것이지 제거되지 않는다.

셋째, Kahn's Algorithm이 구현 복잡도 대비 효과가 크다. 위상 정렬은 잘 알려진 알고리즘이고, 순환 의존 감지까지 한 번의 순회로 처리된다.

## 대안

**고정 순서(executionOrder만)**: 구현이 단순하다. 하지만 병렬 실행이 불가능하고, "A 완료 후 B+C 동시 시작" 같은 패턴을 표현할 방법이 없다. 실무 요구사항을 충족하지 못해 폐기했다.

**워크플로우 엔진 라이브러리 사용 (Temporal, Conductor 등)**: 성숙한 DAG 실행 엔진을 외부 도입하는 방법이다. PoC 목적에서는 외부 의존이 과하고, 학습 비용이 높다. 직접 구현으로 DAG 개념을 검증하는 것이 이 프로젝트의 목표에 더 적합하다.

## 영향

- `pipeline_job_dependency` 테이블이 추가된다
- 파이프라인 실행 엔진에 위상 정렬 로직이 필요하다
- 파이프라인 저장 시 순환 의존 검사 로직이 필요하다
- UI에서 의존 관계를 직관적으로 편집할 수 있는 DAG 에디터가 필요해진다 (Phase 3에서 ReactFlow로 구현)
- 실행 중 각 Job의 상태를 개별 추적해야 하므로 Job 실행 이력 테이블이 필요하다
