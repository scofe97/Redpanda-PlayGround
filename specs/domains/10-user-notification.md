# 사용자 및 알림 도메인

## 개요

사용자 모델과 알림 시스템을 관리하는 도메인이다. 파이프라인/티켓의 생성자·실행자를 추적하는 최소 사용자 모델, 역할 기반 접근 제어(RBAC), 파이프라인 이벤트 알림을 포함한다.

현재 SSE 실시간 알림과 감사 추적(audit)은 구현되어 있다. SSE는 브라우저가 열려 있는 동안 파이프라인 상태를 스트리밍하는 용도이고, 감사 추적은 `playground.audit.events` 토픽에 모든 작업 이력을 기록한다. 사용자 모델과 권한 체계는 미구현이며, 오프라인 알림(웹훅/이메일)도 Phase 4에서 구현 예정이다.

결재는 PoC 범위 외이며, 워크플로우 도메인(02)의 결재 스텁(REQ-02-006)에서 최소한만 다룬다.

## 요구사항

### REQ-10-001: 사용자 기본 모델 [P1] — 미구현

파이프라인/티켓의 소유자와 실행자를 추적하는 최소 사용자 모델을 구현한다. Spring Security를 도입하지 않고 요청 헤더 기반의 단순화된 인증 방식을 사용한다.

**수용 기준**:
- [ ] 사용자 엔티티: ID, 이름, 이메일, 역할(`ADMIN/OPERATOR/VIEWER`)
- [ ] 사용자 초기 데이터는 Flyway 마이그레이션으로 삽입 (PoC)
- [ ] 티켓 생성 시 `createdBy`, 파이프라인 실행 시 `executedBy` 기록
- [ ] 인증: 요청 헤더 `X-User-Id`로 사용자 ID 전달, 인터셉터에서 SecurityContext에 설정
- [ ] 사용자 CRUD API: `POST/GET/PUT/DELETE /api/users`

**구현 참조**: (미구현)

**Phase**: 4

### REQ-10-002: 역할 기반 접근 제어 [P2] — 미구현

역할에 따라 API 접근을 제한하는 기본 구조를 만든다. 세밀한 권한 제어보다는 ADMIN/OPERATOR/VIEWER 세 역할의 명확한 분리를 목표로 한다.

**수용 기준**:
- [ ] ADMIN: 파이프라인/티켓 전체, 미들웨어/환경 생성·수정·삭제, 사용자 관리
- [ ] OPERATOR: 파이프라인/티켓 생성·실행, 미들웨어/환경 조회
- [ ] VIEWER: 파이프라인/티켓/미들웨어/환경 조회만 허용
- [ ] AOP 또는 인터셉터 기반 권한 체크 (`@RequireRole(OPERATOR)` 어노테이션)
- [ ] 권한 부족 시 403 Forbidden 반환

**구현 참조**: (미구현)

**Phase**: 4

### REQ-10-003: 파이프라인 이벤트 알림 [P1] — 미구현

파이프라인 완료/실패 이벤트 발생 시 관련 사용자에게 알림을 발송한다. 브라우저를 닫은 사용자에게도 전달되도록 SSE 외에 웹훅 채널을 추가한다.

**수용 기준**:
- [ ] 알림 이벤트: 파이프라인 완료/실패, 워크플로우 결재 대기, 결재 완료
- [ ] SSE 채널: 기존 SseEmitterRegistry 재사용 (브라우저 접속 중)
- [ ] 웹훅 채널: 설정된 URL로 HTTP POST (Slack/Teams 연동)
- [ ] 이메일 채널: P2 (SMTP 연동)
- [ ] 사용자별 알림 설정: 어떤 이벤트를 어떤 채널로 받을지 선택 가능
- [ ] 기존 `audit.events` 토픽을 알림 소스로 활용 (별도 토픽 불필요)

**구현 참조**: (미구현)

**Phase**: 4

## 설계 결정

### Spring Security 미도입 결정

PoC에서 Spring Security를 도입하면 JWT 발행/검증, 로그인 API, 세션 관리 등 복잡도가 크게 올라간다. 이는 PoC의 핵심 검증 대상인 이벤트 기반 파이프라인과 무관하다. 대신 요청 헤더 `X-User-Id`를 읽는 인터셉터로 사용자를 식별하고, AOP로 권한을 체크하는 방식을 사용한다. 프로덕션 전환 시 헤더 인증을 JWT로 교체하면 인터셉터만 수정하면 된다.

### SSE와 알림의 관계

SSE는 "진행 중인 작업의 상태 스트림"이고, 알림은 "완료/실패 같은 이벤트의 통지"로 개념이 다르다. 두 가지를 같은 인프라(SSE)로 전달할 수 있지만, 브라우저를 닫은 사용자에게는 SSE가 전달되지 않으므로 웹훅/이메일 채널이 별도로 필요하다. `NotificationAdapter` 인터페이스로 두 채널을 통합 관리하되, SSE는 기존 `SseEmitterRegistry`를 재사용한다.

### 알림 소스: 새 토픽 vs 기존 audit.events 재활용

알림 발행을 위해 별도 토픽을 만드는 대신 기존 `playground.audit.events` 토픽의 컨슈머로 구현한다. 이미 모든 작업 이력이 이 토픽에 발행되므로, 알림 컨슈머가 이를 구독하여 이벤트 유형에 따라 알림을 선택적으로 발송하면 된다. 새 토픽을 추가하지 않아 인프라 복잡도를 줄일 수 있다.

## 미해결 사항

- **사용자 모델 없이 추적 불가 (REQ-10-001)**: 현재 파이프라인/티켓 생성자와 실행자를 기록하지 않아 "누가 이 파이프라인을 실행했는가"를 알 수 없다. RBAC 구현에 앞서 사용자 모델이 선행되어야 한다.
- **오프라인 알림 경로 없음 (REQ-10-003)**: 브라우저를 닫은 상태에서 파이프라인이 완료되면 알림을 받을 방법이 없다. 웹훅 채널 구현이 가장 우선순위가 높다.
- **결재 연계 (REQ-02-006)**: 결재 스텁이 "결재 대기" 상태를 발생시키면 알림 인프라가 동작해야 하지만, 사용자 모델 없이는 "누구에게" 알림을 보낼지 결정할 수 없다.
