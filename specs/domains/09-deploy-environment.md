# 배포 환경 도메인

## 개요

배포 환경은 배포 대상(VM 서버, K8S 클러스터)을 관리하는 도메인이다. Deploy Job(06)이 "어디에 배포할지"를 결정할 때 이 도메인의 엔티티를 참조한다. 미들웨어(08)가 "어떤 도구로" 배포할지를 관리한다면, 배포 환경(09)은 "어디에" 배포할지를 관리한다. 두 도메인은 직접 참조하지 않고, 배포 Job 실행 시점에 합류한다.

환경 엔티티는 접속 정보(SSH 키, kubeconfig)를 평문으로 저장하지 않는다. 인증 정보는 Credential 엔티티에 분리하여 암호화 저장하고, 환경 엔티티는 `credentialId`로 간접 참조한다. Phase 4에서 구현 예정이다.

## 요구사항

### REQ-09-001: VM 서버 엔티티 [P0] — 미구현

VM 서버 정보를 관리하는 엔티티를 정의한다. 접속 인증 정보는 평문 저장을 금지하고 Credential 엔티티로 분리한다.

**수용 기준**:
- [ ] 서버 ID, 이름, 설명, 호스트(IP 또는 도메인), SSH 포트
- [ ] 인증 방식: SSH 키 / 패스워드 선택
- [ ] 인증 정보는 `credentialId`로 간접 참조 (평문 저장 금지)
- [ ] 배포 기본 경로 (`deployBasePath`)
- [ ] 상태: `ACTIVE / INACTIVE`
- [ ] 태그/라벨 (환경 구분: dev/staging/prod)

**구현 참조**: (미구현)

**Phase**: 4

### REQ-09-003: 환경 CRUD [P0] — 미구현

VM 서버와 K8S 클러스터를 통합 관리하는 REST API를 제공한다. 등록/조회/수정/삭제 및 연결 테스트를 포함한다.

**수용 기준**:
- [ ] VM 서버 CRUD: `POST/GET/PUT/DELETE /api/environments/vm`
- [ ] K8S 클러스터 CRUD: `POST/GET/PUT/DELETE /api/environments/k8s`
- [ ] 통합 목록 조회: 환경 유형(VM/K8S) 필터, 태그 필터
- [ ] 환경 연결 테스트: VM은 SSH 접속 테스트, K8S는 API 서버 접근 테스트
- [ ] 연결 테스트 결과를 상태 필드에 반영

**구현 참조**: (미구현)

### REQ-09-004: 배포 Job 연계 [P1] — 미구현

배포 Job(06)이 `environmentId`로 환경 엔티티를 참조하여 배포 대상 접속 정보를 해석한다. 파이프라인 실행 시점에 environmentId로 환경 엔티티를 조회하고, 유형에 따라 VM/K8S 배포 전략을 선택한다.

**수용 기준**:
- [ ] 배포 Job config에 `environmentId` 필드 추가
- [ ] 파이프라인 실행 시 `environmentId` → 환경 엔티티 조회 → 접속 정보 해석
- [ ] VM 배포: 서버 SSH 정보(호스트, 포트, credentialId) 사용
- [ ] K8S 배포: 클러스터 API 서버 URL + 인증 정보 사용
- [ ] 환경이 `INACTIVE` 상태일 때 배포 실행 거부 (400 에러)

**구현 참조**: (미구현)

**Phase**: 4

### REQ-09-005: Credential 엔티티 [P2] — 미구현

환경 접속에 필요한 인증 정보를 안전하게 저장하고 관리한다. 환경 엔티티는 `credentialId`로만 참조하여 인증 값을 직접 노출하지 않는다.

**수용 기준**:
- [ ] Credential 엔티티: ID, 이름, 타입(`SSH_KEY/PASSWORD/TOKEN/KUBECONFIG`), 값(암호화)
- [ ] 값은 AES-256으로 암호화 저장, 암호화 키는 환경변수에서 로드
- [ ] 조회 API에서 값을 마스킹하여 반환 (노출 금지)
- [ ] 환경 엔티티에서 `credentialId`로 참조

**구현 참조**: (미구현)

**Phase**: 4

## 설계 결정

### VM 서버와 K8S 클러스터를 통합 엔티티로 관리

두 가지 접근이 가능하다. 통합 엔티티 방식은 `Environment(type: VM/K8S)` + config JSON으로 유형별 정보를 구분하여 단순하지만, config 스키마가 유형마다 달라 타입 안전성이 낮다. 분리 엔티티 방식은 `VmServer`, `K8sCluster`를 별도 테이블로 두어 타입 안전하지만 API가 분리된다.

PoC에서는 통합 엔티티로 시작한다. `environmentType: VM | K8S`와 config JSON으로 구분하되, config의 필수 필드를 유형별로 서비스 레이어에서 검증한다. 규모가 커지면 분리 엔티티로 전환한다.

### 인증 정보 보안 수준

PoC이므로 HashiCorp Vault 수준의 연동은 불필요하지만 평문 저장은 피해야 한다. DB에 AES-256으로 암호화 저장하고, 키는 환경변수에서 로드하는 수준이 적절하다. Credential 엔티티를 분리해두면 추후 Vault나 AWS Secrets Manager로 교체할 때 환경 엔티티를 변경하지 않아도 된다.

### 환경과 프리셋의 관계

미들웨어 프리셋은 "도구 조합"이고, 배포 환경은 "배포 목적지"다. ArgoCD 정보는 프리셋에서, K8S 클러스터 접속 정보는 환경에서 각각 조회한다. 두 도메인은 직접 참조하지 않고 배포 Job 실행 시점에 합류한다.

## 미해결 사항

- **환경 엔티티 전체 미구현**: 배포 Job(06)의 VM/K8S 분기(REQ-06-001)가 환경 도메인 구현에 선행 의존한다. 두 도메인을 함께 구현해야 E2E 배포 흐름이 완성된다.
- **Credential 암호화 구현**: AES-256 암호화 키 관리 방식을 결정해야 한다. 환경변수 로드 방식으로 시작하되, 키 로테이션 정책이 없으면 장기적으로 취약하다.
- **연결 테스트 API (REQ-09-003)**: SSH 접속 테스트와 K8S API 서버 접근 테스트를 동기 호출로 구현하면 타임아웃 처리가 필요하다. 비동기 테스트 후 상태를 폴링하는 방식도 검토해야 한다.
