# 02. 워크플로우 도메인 스펙

## 개요

워크플로우는 업무코드별 프로세스 템플릿이다. 빌드(BUILD), 배포(DEPLOY), 결재(APPROVAL) 컴포넌트를 조합하여 "무엇을 어떤 순서로 진행할지"를 정의한다. 파이프라인이 "어떻게 실행할지(Job 순서)"를 담당한다면, 워크플로우는 그 위에서 비즈니스 프로세스 흐름을 정의하는 상위 개념이다.

Phase 5에서 전체 구현 예정이다. 현재는 미구현 상태이며, 티켓이 파이프라인을 직접 트리거하는 레거시 구조가 대신 동작하고 있다.

## 요구사항

### REQ-02-001: 워크플로우 엔티티 정의 [P0]

워크플로우를 정의하는 엔티티를 설계한다. 컴포넌트 목록은 순서가 있는 리스트로 관리한다.

**수용 기준**
- [ ] 워크플로우 ID, 이름, 설명 필드 존재
- [ ] 컴포넌트 목록(순서 있는 리스트)을 별도 테이블로 관리
- [ ] 활성 상태 `ACTIVE / INACTIVE` 지원
- [ ] 생성/수정 시간 기록

---

### REQ-02-002: 컴포넌트 타입 정의 [P0]

워크플로우를 구성하는 컴포넌트 타입을 정의한다. 타입마다 참조 대상이 다르다.

| 컴포넌트 타입 | 참조 대상 | 설명 |
|-------------|----------|------|
| BUILD | 파이프라인 ID | 빌드 파이프라인 실행 |
| DEPLOY | 파이프라인 ID | 배포 파이프라인 실행 |
| BUILD_AND_DEPLOY | 파이프라인 ID | 빌드+배포 통합 파이프라인 |
| APPROVAL | 승인자 정보 | 결재 대기 (PoC: 자동 승인 스텁) |

**수용 기준**
- [ ] 컴포넌트별 `order` 필드로 실행 순서 결정
- [ ] 하나의 워크플로우에 동일 타입 컴포넌트 복수 허용 (예: BUILD → APPROVAL → DEPLOY)
- [ ] 각 컴포넌트는 독립 상태(PENDING / IN_PROGRESS / COMPLETED / FAILED / SKIPPED) 보유

---

### REQ-02-003: 워크플로우 CRUD [P0]

워크플로우를 관리하는 API를 제공한다.

**수용 기준**
- [ ] 워크플로우 생성: 이름 + 컴포넌트 리스트
- [ ] 워크플로우 조회: 상세(컴포넌트 포함) / 목록
- [ ] 워크플로우 수정: 컴포넌트 추가/제거/순서 변경
- [ ] 워크플로우 삭제: 참조 중인 업무코드가 있으면 삭제 불가 (409 반환)

---

### REQ-02-004: 업무코드-워크플로우 매핑 [P0]

업무코드와 워크플로우의 매핑을 관리한다.

**수용 기준**
- [ ] 업무코드 1개 → 워크플로우 1개 (1:1 매핑)
- [ ] 매핑 생성 / 조회 / 수정 / 삭제 API
- [ ] 매핑이 없는 업무코드는 "기본 워크플로우" 적용
- [ ] 기본 워크플로우는 시스템 설정으로 관리

---

### REQ-02-005: 워크플로우 실행 엔진 [P1]

워크플로우를 실행하는 엔진을 구현한다. 컴포넌트를 순서대로 처리하며, 각 컴포넌트의 완료를 이벤트로 수신한 뒤 다음 컴포넌트로 진행한다.

**수용 기준**
- [ ] BUILD/DEPLOY 컴포넌트: 해당 파이프라인 실행 트리거 → 완료 이벤트 수신 후 다음 단계 진행
- [ ] APPROVAL 컴포넌트: 결재 대기 상태 전환 → 승인/거절 이벤트 수신
- [ ] 컴포넌트 실패 시 워크플로우 전체를 FAILED로 전환
- [ ] 실행 중 동기 블로킹 없음 (이벤트 기반 비동기 진행)

---

### REQ-02-006: 결재 스텁 [P2]

실제 결재 시스템 없이 워크플로우의 결재 컴포넌트가 동작하는 구조를 스텁으로 검증한다.

**수용 기준**
- [ ] 워크플로우 실행 중 APPROVAL 컴포넌트 도달 시 결재 요청 레코드 생성
- [ ] 수동 승인 API (`POST /approvals/{id}/approve`) 및 거절 API (`/reject`) 제공
- [ ] PoC 옵션: 자동 승인 모드 (환경 설정으로 on/off)
- [ ] 승인 → 워크플로우 다음 단계 진행, 거절 → 워크플로우 REJECTED 상태 전환

---

### REQ-02-007: 워크플로우 실행 이력 [P2]

워크플로우 실행 내역을 추적한다.

**수용 기준**
- [ ] 실행 레코드: 워크플로우 실행 ID, 티켓 ID, 워크플로우 ID
- [ ] 각 컴포넌트 실행 상태 + 시작/종료 시간 기록
- [ ] BUILD/DEPLOY 컴포넌트는 파이프라인 실행 ID와 연결
- [ ] 이력 조회 API (`GET /workflow-executions/{id}`)

## 설계 결정

### 워크플로우 엔진은 파이프라인 API를 단방향으로 호출

파이프라인은 워크플로우를 모른다. 워크플로우 엔진이 파이프라인 실행 API를 호출하고, 파이프라인 완료 이벤트를 수신하여 다음 컴포넌트로 진행하는 단방향 의존이다. 이를 뒤집으면 파이프라인의 독립 실행(UC-1)이 불가능해진다.

### 결재는 워크플로우 레벨, 파이프라인 레벨이 아님

파이프라인에 결재를 넣으면 "테스트 환경은 결재 없이, 프로덕션은 결재 후 실행"하는 유연성이 사라진다. 결재는 비즈니스 프로세스의 승인 절차이므로 워크플로우 레벨에서 처리하고, 파이프라인은 실행 기계로서의 역할만 유지한다.

### 워크플로우 없는 직접 실행은 허용

UC-1처럼 파이프라인을 직접 실행하는 경로는 워크플로우를 거치지 않는다. 워크플로우는 "티켓 기반 업무 프로세스"가 필요한 경우에만 개입한다.

## 도메인 간 관계

```
BusinessCode (01-ticket)
  └─ WorkflowMapping → Workflow
       └─ Component[BUILD]  → Pipeline (03-pipeline)
       └─ Component[DEPLOY] → Pipeline (03-pipeline)
       └─ Component[APPROVAL] → ApprovalRequest

WorkflowExecution
  └─ ticketId (01-ticket)
  └─ workflowId
  └─ ComponentExecution[]
       └─ pipelineExecutionId (BUILD/DEPLOY 타입)
```

- 티켓 도메인: `01-ticket.md`
- 파이프라인 도메인: `03-pipeline.md`
