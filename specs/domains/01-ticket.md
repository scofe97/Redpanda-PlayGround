# 01. 티켓 도메인 스펙

## 개요

티켓은 배포 업무 요청의 시작점이다. 각 티켓에는 업무코드(Business Code)가 부여되고, 업무코드가 워크플로우를 결정한다. 워크플로우는 빌드/배포/결재 등 컴포넌트 조합으로 구성되며, 컴포넌트가 파이프라인을 참조하여 실행한다.

현재는 기본 CRUD만 구현되어 있고, 업무코드 기반 워크플로우 연계는 Phase 5에서 구현 예정이다. 레거시 구조(티켓→파이프라인 직접 실행)는 Phase 1 독립화 작업에서 정리될 예정이다.

## 현재 상태 (2026-03-22)

| 항목 | 상태 | 비고 |
|------|------|------|
| Ticket CRUD | 부분 구현 | 기본 생성/조회만. 수정/삭제 미검증 |

| 파이프라인 독립성 | 미구현 | 파이프라인이 티켓에 종속된 구조 |

## 요구사항

### REQ-01-001: 업무코드 필드 추가 [P0]

티켓 엔티티에 `businessCode` 필드를 추가한다. 업무코드는 팀 단위 식별자로, 티켓 생성 시 입력하면 해당 팀의 워크플로우가 자동으로 결정된다.

**수용 기준**
- [ ] 티켓 엔티티에 `businessCode` 필드(String) 추가
- [ ] 업무코드 목록을 관리하는 `BusinessCode` 엔티티 별도 존재
- [ ] 티켓 생성 API에서 유효한 업무코드인지 검증 (존재하지 않으면 400 반환)
- [ ] 업무코드는 PoC 초기에 optional, 워크플로우 연계 안정화 후 required로 전환

예시 업무코드: `"TEAM-A"`, `"PLATFORM"`, `"INFRA"`

---

### REQ-01-002: 워크플로우 참조 [P0]

티켓 생성 시 업무코드를 기반으로 워크플로우를 자동 결정하는 구조를 만든다. "어떤 프로세스를 따를지"가 업무코드로부터 자동으로 해석된다.

**수용 기준**
- [ ] 업무코드 1개는 워크플로우 1개에 매핑 (1:1)
- [ ] 매핑이 없는 업무코드는 기본 워크플로우(빌드+배포만) 적용
- [ ] 티켓 조회 응답에 연결된 워크플로우 정보 포함

---

### REQ-01-003: 파이프라인 독립성 확보 [P0]

기존의 "티켓 생성 → 파이프라인 자동 생성" 종속 관계를 끊는다. 파이프라인은 티켓 없이도 존재하고 실행될 수 있어야 한다.

**수용 기준**
- [ ] 티켓 생성이 파이프라인 생성을 자동 트리거하지 않음
- [ ] 워크플로우의 빌드/배포 컴포넌트가 기존 파이프라인을 참조
- [ ] 파이프라인 단독 실행 API는 티켓 없이 동작 (UC-1 유지)

---

### REQ-01-004: 업무코드 CRUD [P1]

업무코드를 관리하는 API를 제공한다.

**수용 기준**
- [ ] 업무코드 생성 / 조회 / 수정 / 삭제
- [ ] 업무코드별 기본 워크플로우 매핑 설정
- [ ] 업무코드별 기본 미들웨어 프리셋 매핑 설정 (편의 기능)

---

### REQ-01-005: 티켓 상태 확장 [P1]

워크플로우 도입에 따라 티켓 상태를 확장한다.

**수용 기준**
- [ ] 상태 전이: `DRAFT → READY → WORKFLOW_IN_PROGRESS → DEPLOYING → APPROVAL_PENDING → DEPLOYED / FAILED / REJECTED`
- [ ] `WORKFLOW_IN_PROGRESS`: 워크플로우 실행 중 (빌드/배포 컴포넌트 처리 중)
- [ ] `APPROVAL_PENDING`: 결재 대기 (워크플로우에 APPROVAL 컴포넌트가 있을 때)
- [ ] `REJECTED`: 결재 거절로 인한 최종 실패

---

### REQ-01-006: 티켓-파이프라인 실행 이력 [P2]

티켓이 트리거한 파이프라인 실행 내역을 추적한다. 재시도 시나리오를 포함한다.

**수용 기준**
- [ ] 하나의 티켓이 여러 번 실행 이력을 가질 수 있음 (재시도)
- [ ] 이력 항목: 파이프라인 ID, 실행 결과(SUCCESS/FAILED), 시작/종료 시간
- [ ] 이력 조회 API (`GET /tickets/{id}/executions`)

## 설계 결정

### 업무코드-워크플로우 1:1 매핑으로 시작

TPS와 동일하게 1:1로 시작한다. "같은 팀이지만 상황에 따라 다른 프로세스"가 필요해지면 1:N으로 확장할 수 있도록 매핑 테이블을 별도 엔티티로 분리한다. 직접 FK로 박지 않는다.

### 파이프라인 추적은 Execution 레벨에서

파이프라인 엔티티 자체에 티켓 참조를 두지 않는다. 실행(`PipelineExecution`) 레코드에 `triggerSource` (TICKET / MANUAL / WORKFLOW) + `triggerId`를 기록하는 방식으로 추적한다. 파이프라인의 독립성이 유지되고, 역방향 조회도 가능하다.

### 업무코드 optional로 시작

기존 기능(티켓→파이프라인 직접 실행)이 깨지지 않도록 PoC 초기에는 `businessCode`를 optional로 운영한다. 워크플로우 연계 완료 후 마이그레이션 스크립트로 required 전환한다.

## 도메인 간 관계

```
Ticket
  └─ businessCode → BusinessCode (1:1)
       └─ WorkflowMapping → Workflow (02-workflow)
            └─ Component[BUILD] → Pipeline (03-pipeline)
            └─ Component[DEPLOY] → Pipeline (03-pipeline)
            └─ Component[APPROVAL] → ApprovalStub

PipelineExecution
  └─ triggerSource: TICKET | MANUAL | WORKFLOW
  └─ triggerId: ticketId (TICKET일 때)
```

- 워크플로우 도메인: `02-workflow.md`
- 파이프라인 도메인: `03-pipeline.md`

---

## 미해결 사항

- 업무코드 없이 생성된 기존 티켓의 마이그레이션 전략 미정
- APPROVAL_PENDING 상태에서 타임아웃 처리 정책 미결정
- 하나의 티켓이 여러 워크플로우를 순차 실행해야 하는 케이스 발생 가능성 검토 필요
