Transactional Outbox 아키텍처

common-kafka 모듈 — DB 트랜잭션과 메시지 발행의 원자성을 보장하는 폴링 기반 Outbox 패턴 구현

개요

Transactional Outbox 패턴은 비즈니스 로직의 DB 변경과 Kafka 메시지 발행을 하나의 원자적 단위로 만드는 설계 기법이다. 비즈니스 트랜잭션 내에서 Kafka로 직접 발행하면, DB 커밋은 성공했지만 Kafka 발행이 실패하는 "이중 쓰기(Dual Write)" 문제가 발생할 수 있다.

이 모듈은 이벤트를 DB의 outbox_event 테이블에 먼저 저장하고, 별도 폴러가 주기적으로 이를 읽어 Kafka로 발행하는 방식으로 이 문제를 해결한다. DB 트랜잭션이 롤백되면 outbox 레코드도 함께 롤백되므로, "메시지가 발행됐지만 비즈니스 로직은 실패한" 상황이 원천 차단된다.

왜 필요한가

이중 쓰기 문제
// 위험한 패턴: DB 커밋 후 Kafka 발행 실패 가능
@Transactional
public void createJob(JobCommand cmd) {
    jobRepository.save(job);          // ✅ DB 커밋 성공
    kafkaTemplate.send(topic, event); // ❌ 브로커 다운 → 메시지 유실
}

// Outbox 패턴: 같은 TX에서 이벤트를 DB에 저장
@Transactional
public void createJob(JobCommand cmd) {
    jobRepository.save(job);
    eventPublisher.publish(             // ✅ 같은 TX → 원자성 보장
        "EXECUTION_JOB", executionId
        , "JOB_EXECUTE"
        , avroSerializer.serialize(event)
        , Topics.EXECUTOR_CMD_JOB_EXECUTE, executionId
    );
}

구성 요소

JPA Entity
OutboxEvent
이벤트 저장소. aggregateType, aggregateId, eventType, payload(Avro 바이너리), status, retryCount, traceParent 등을 보관한다. 팩토리 메서드 OutboxEvent.of()로 생성한다.
Publisher (진입점)
EventPublisher
비즈니스 코드가 호출하는 유일한 진입점이다. AvroSerializer로 직렬화하고, TraceContextUtil로 현재 traceparent를 캡처한 뒤 outbox_event 테이블에 INSERT한다. 반드시 @Transactional 컨텍스트 안에서 호출해야 한다.
Poller (핵심 엔진)
OutboxPoller
500ms 주기로 PENDING 이벤트를 SELECT ... FOR UPDATE SKIP LOCKED로 가져온다. CloudEvents 헤더 추가, traceparent 복원, Kafka 발행, 실패 시 지수 백오프 재시도, 최종 실패 시 DEAD 전환까지 전체 발행 라이프사이클을 관리한다. Aggregate 단위 순서를 보장한다.
Repository
OutboxRepository
SKIP LOCKED 기반 CTE 쿼리로 동시 폴러 간 경쟁을 방지한다. batchMarkAsSent(), incrementRetryAndSetNextRetryAt(), markAsDead(), deleteOlderThan() 등 상태 전이를 위한 네이티브 SQL을 제공한다.
직렬화
AvroSerializer
Confluent Wire Format(0x00 + 4바이트 schema ID + Avro binary)으로 직렬화한다. Schema Registry에 스키마를 자동 등록하고, RecordNameStrategy로 subject를 결정한다. JSON 역직렬화(deserializeJson)도 지원하여 Jenkins 같은 외부 시스템의 JSON 페이로드를 처리한다.
트레이싱
TraceContextUtil
OpenTelemetry가 classpath에 있으면 W3C traceparent를 캡처/복원한다. OTel이 없으면 자동으로 건너뛴다(graceful degradation). 내부의 OtelBridge가 Span 생성, 부모 컨텍스트 복원, 속성 기록을 처리한다.
설정
OutboxProperties
outbox.* prefix로 바인딩된다. batchSize(50), pollIntervalMs(500), maxRetries(5), cleanupRetentionDays(7), cleanupCron(매일 03시)을 제공한다.
관측성
OutboxMetrics
Micrometer 기반 4개 메트릭을 등록한다. Counter: outbox.events.published, outbox.events.failed, outbox.events.dead. Gauge: outbox.queue.pending(현재 PENDING 수).

이벤트 라이프사이클

모든 outbox 이벤트는 4가지 상태를 거친다. PENDING에서 시작하여 정상 경로에서는 SENT로, 반복 실패 시 DEAD로 종료된다.

상태 머신
PENDING
→
Poll SELECT
PROCESSING
→
Kafka OK
SENT
PROCESSING에서 실패 시: retryCount < maxRetries → PENDING (재시도) retryCount ≥ maxRetries → DEAD

PROCESSING → PENDING 복귀 시 revertToRetryable()가 호출된다. 이 메서드는 WHERE status = 'PROCESSING' 조건이 있어, 이미 다른 경로로 상태가 변경된 이벤트를 실수로 되돌리는 것을 방지한다.

지수 백오프

실패한 이벤트는 2^retryCount초 만큼 대기 후 재시도된다. 지수적으로 간격이 늘어나므로 일시적 장애 시 브로커에 과도한 부하를 주지 않으면서, 짧은 장애에는 빠르게 복구할 수 있다.

// OutboxPoller.handleFailure()
if (event.getRetryCount() >= properties.getMaxRetries()) {
    repository.markAsDead(event.getId());
    metrics.incrementDead();
} else {
    repository.revertToRetryable(event.getId());
    LocalDateTime nextRetry = LocalDateTime.now()
        .plusSeconds((long) Math.pow(2, event.getRetryCount()));
    repository.incrementRetryAndSetNextRetryAt(event.getId(), nextRetry);
    metrics.incrementFailed();
}

트레이싱

Outbox 패턴은 이벤트를 비동기로 발행하기 때문에, 원래 요청의 트레이스 컨텍스트가 유실될 수 있다. 이 모듈은 EventPublisher.publish() 시점에 현재 OTel Span의 traceparent를 outbox_event.trace_parent 컬럼에 저장하고, OutboxPoller가 Kafka 발행 시 이를 복원하여 부모 Span으로 설정한다.

traceparent 흐름
① 저장 시점 (EventPublisher)
HTTP 요청 Span → TraceContextUtil.captureTraceParent()
                → "00-{traceId}-{spanId}-01"
                → outbox_event.trace_parent 컬럼에 저장

② 발행 시점 (OutboxPoller)
outbox_event.trace_parent 읽기
→ TraceContextUtil.publishWithTrace()
  → SpanContext.createFromRemoteParent(traceId, spanId)
  → tracer.spanBuilder("OutboxPoller.publish").setParent(...)
  → Span 속성: outbox.event.id, outbox.event.type, outbox.aggregate.id
  → kafkaTemplate.send() 실행
  → traceparent 헤더가 Kafka 레코드에 자동 전파

OpenTelemetry가 classpath에 없으면 TraceContextUtil.isOtelAvailable()이 false를 반환하고, 트레이싱 관련 코드를 전부 건너뛴다. 런타임 에러 없이 정상 동작한다.

CloudEvents 헤더

모든 Outbox 이벤트는 CloudEvents 1.0 Binary Content Mode 헤더를 포함한다. 이 표준 헤더를 통해 소비자가 메시지의 출처, 유형, 상관 관계를 메타데이터만으로 파악할 수 있다.

헤더값설명
ce_specversion"1.0"CloudEvents 스펙 버전
ce_id이벤트 ID (PK)메시지 고유 식별자
ce_source"/{applicationName}"발행 서비스 식별 (spring.application.name)
ce_typeeventType이벤트 유형 (JOB_EXECUTE, JOB_STARTED_NOTIFY 등)
ce_correlationidcorrelationId요청 추적 ID (선택)
traceparentW3C formatOTel 트레이스 전파 (선택)

CloudEventsHeaderInterceptor가 KafkaTemplate의 ProducerInterceptor로 등록되어 기본 헤더(ce_specversion, ce_id, ce_source, ce_time)를 자동 추가한다. OutboxPoller는 이벤트별 헤더(ce_type, ce_correlationid)를 추가로 설정한다.

테이블 스키마

CREATE TABLE IF NOT EXISTS outbox_event (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,
    aggregate_type  VARCHAR(100)  NOT NULL,
    aggregate_id    VARCHAR(255)  NOT NULL,
    event_type      VARCHAR(100)  NOT NULL,
    payload         LONGBLOB      NOT NULL,     -- Avro wire format
    topic           VARCHAR(255)  NOT NULL,
    status          VARCHAR(20)   NOT NULL DEFAULT 'PENDING',
    created_at      DATETIME(6)   NOT NULL DEFAULT CURRENT_TIMESTAMP(6),
    sent_at         DATETIME(6)   NULL,
    retry_count     INT           NOT NULL DEFAULT 0,
    correlation_id  VARCHAR(255)  NULL,
    trace_parent    VARCHAR(255)  NULL,
    next_retry_at   DATETIME(6)   NULL,

    INDEX idx_outbox_pending (status, next_retry_at),  -- Poll 쿼리
    INDEX idx_outbox_cleanup (status, sent_at)          -- Cleanup 쿼리
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

idx_outbox_pending은 Poller의 SELECT ... WHERE status='PENDING' AND next_retry_at <= NOW() 쿼리를 커버한다. idx_outbox_cleanup은 deleteOlderThan()의 SENT 레코드 정리 쿼리를 커버한다.
