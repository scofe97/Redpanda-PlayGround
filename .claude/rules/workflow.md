# 작업 워크플로우 규칙

## 요구사항 명확화

- 요청에서 **확정/불명확/가정** 분류 후 핵심 질문 (최대 5개)
- 5W1H: What(범위), Who(사용자), Where(위치), When(시점), Why(목적), How(방식)
- 답변 바탕으로 기능 요약, 상세 요구사항, 제약 조건 확정
- 불확실하면 추측으로 진행하지 않음 -- 질문

## 자율 실행 조건

**적합**: 반복 패턴 작업, 규칙 기반 작업, 검증/롤백 가능한 작업
**부적합**: 비즈니스 로직 변경, 아키텍처 결정, 외부 API 연동, DB 수정

탈출 조건: 동일 오류 3회, 빌드 실패, 작업량 2배 초과

## 컨텍스트 관리 (필수)

- 도메인 단위로 필요한 코드만 제공 -- AI 집중도 상승, 환각 감소
- **파일 5개 이상** 읽기/쓰기 -- 백그라운드 에이전트에 위임
- **도구 호출 20회 도달** -- `/compact` 실행

### 작업 분할 전략
- 10개 이상 파일 작업 -- 2~3개 배치로 나눠서 진행
- 각 배치 완료 후 `/compact` -- 다음 배치 진행

## 목표 주도 실행

- 모호한 요청 -- 구체적 **완료 조건** 정의 후 시작
- 예: "잘 동작하게 해줘" -- "X 입력 시 Y 출력, Z 에러 없음"
- 각 단계마다 목표에 가까워지고 있는지 자기 점검
- 완료 시 정의한 완료 조건 대비 결과 검증

## 스펙 기반 개발

1. 새 기능 구현 전 해당 도메인 스펙(`specs/domains/`)의 **수용 기준** 확인
2. 구현 완료 시 수용 기준 체크박스를 `[x]`로 업데이트
3. Phase 작업 시 해당 phase 스펙(`specs/phases/`)을 먼저 로드
4. 아키텍처 변경 시 `specs/decisions/`에 ADR 추가
